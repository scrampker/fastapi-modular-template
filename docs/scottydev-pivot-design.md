# ScottyDev Pivot — Architecture Design v4

> **Status:** Proposed (v4 — hardened isolation, AI-required connector wizard, user-provided master key)
> **Revision date:** 2026-04-16 v4
> Prior constraints carried forward from v2:
> - **Constraint A** — Three-level recursive forkability (framework / core / project)
> - **Constraint B** — Per-workspace feature opinionation (two workspaces, one install, different feature sets)
> New directives applied in v3:
> - **Directive 1** — Pluggable infrastructure connector system replaces vendor-scottylab_toolkit
> - **Directive 2** — Workspace is a UI-created database concept, not filesystem-first
> - **Directive 3** — scottycore repo name stays; ScottyDev is a distinct platform; repo split recommended
> New directives applied in v4:
> - **Directive 4** — Master key generated by setup wizard, persisted to `.secrets/`, not auto-generated
> - **Directive 5** — Generic connector naming throughout; connector setup is AI-REQUIRED with no click-through default
> - **Directive 6** — Per-workspace OS-level and database-level isolation; zero cross-workspace leakage
> - **Directive 7** — Phase 2/3 AI chat tool scope split formally documented in Q18

---

## Executive Summary

ScottyDev is an active orchestration platform that users log into to create workspaces, connect AI backends and infrastructure, manage apps, and make changes to shared framework code by talking with AI. It is not a read-only dashboard.

V4 makes four structural changes on top of v3:

1. **Hardened per-workspace isolation.** The v3 single-database model (logical `workspace_id` partitioning) is replaced by a database-per-workspace model backed by dedicated Linux UIDs on the ScottyDev host. Cross-workspace leakage is prevented at both the OS and Postgres levels. The ScottyDev process holds a narrow admin credential for workspace CRUD only; all connector and application logic runs as the workspace's own UID.

2. **AI-required connector setup wizard.** There is no factory-default path for connector configuration. Every workspace connector — whether the user has existing automation or not — is configured through an AI-guided conversational wizard. The wizard uses the workspace's configured AI backend. Built-in generic connectors (`DockerSSHConnector`, `ProxmoxSSHConnector`, `SystemdConnector`) are the AI's toolbox; it selects and parameterises one on behalf of the user, or generates a custom stub.

3. **User-managed master key.** `WORKSPACE_ENCRYPTION_KEY` is generated once by the install setup wizard, displayed to the admin, persisted to `.secrets/workspace_encryption_key` (mode 0600) on the ScottyDev host, and gitignored. ScottyDev refuses to start if the key file is absent. The admin owns recovery.

4. **Phase 2/3 AI chat tool scope formally documented.** Q18 defines exactly which AI chat tools ship in Phase 2 (read-only) and which are added in Phase 3 (write tools with confirmation UX), including the `scottydev_pending_confirmations` table schema.

## Q1 — What problem does the ScottyDev pivot solve, and what is ScottyDev vs ScottyCore? *(unchanged from v3)*

### The problem

ScottyCore started as a template to be cloned. Over time it accumulated coordination tasks that are not application domain logic: propagating bug fixes across five consumer apps, reconciling pattern drift, routing AI requests, running manager agents. Those tasks have no natural home in any single app. ScottyDev gives them one.

The pivot reframes the ecosystem from "template you clone" to "platform you run." The template artifacts remain (schemas, service skeletons, middleware); ScottyDev adds the orchestration layer on top.

V3 extended the scope further: ScottyDev is not just a dashboard or a sync watcher. It is the platform from which a workspace admin creates apps, talks with AI to change the shared framework, tests deployments, and monitors the ecosystem — all from a browser.

### The naming

Two distinct things exist:

| Name | What it is | Who uses it |
|---|---|---|
| **scottycore** | A Python package (`pip install scottycore`). Provides FastAPI service modules, auth chain, RBAC, settings KV, audit log, AI backends. Versioned via pyproject.toml. | Every app that depends on the framework — six currently. Import path stays `from scottycore...`. |
| **ScottyDev** | A web application / platform. Users log in, create workspaces, configure connectors and AI backends, manage apps, and orchestrate changes. Lives in a new `scottydev` repo. | Operators who want orchestration across their app fleet. |

In this user's setup, the "shared framework repo" for the ScottyLand workspace is `scottycore`. In AcmeCorp's setup it would be `acme-core`. ScottyDev does not care what the framework repo is called; the workspace admin declares it.

### Repo split recommendation: Shape C

**Recommendation: Shape C — keep `scottycore` as the Python package repo, create a new `scottydev` repo for the platform.**

Rationale in one sentence: the six apps that currently do `pip install scottycore` must not be disrupted, and ScottyDev as a product has a completely different lifecycle (it is deployed once per organisation, not imported into every app).

Shape A couples the product lifecycle to the library version. Shape B leaves orchestration scripts polluting the library repo. Shape C performs a one-time surgery: move `scottycore/meta/`, the `scripts/scottycore-init.py` orchestration layer, and the `.claude/agents/` manager agents into the new `scottydev` repo. The `scottycore` package stays focused on what apps import.

**The `scottycore` pip import path survives Shape C unchanged.** Six apps continue to `from scottycore.services.auth import ...` without modification.

## Q2 — What is a workspace, and how does it differ from an app-level tenant? *(unchanged from v3)*

Two distinct concepts share the word "tenant" in this codebase.

| Concept | Table | Managed by | Scope |
|---|---|---|---|
| **App workspace** (code symbol: `tenant`) | `tenants` | `TenantsService` | Per-app user isolation — unchanged |
| **ScottyDev workspace** | admin DB: `scottydev_workspaces` | `WorkspaceService` | Per-organisation orchestration config in ScottyDev |

The `tenants` table and `TenantsService` are **not modified**. All six consumer apps depend on them. The symbol `tenant` is untouched in all consumer app code.

The `scottydev_workspaces` table lives in the `scottydev` admin Postgres database (see Q3/Q6 for the isolation model). A ScottyDev workspace is identified by a slug (e.g., `scotty-land`, `brians-projects`). Workspaces are created by a logged-in ScottyDev user with the `admin` role or higher.

## Q3 — How is each workspace configured? Database-first model with per-workspace isolation. *(rewritten for v4)*

V2 proposed a `tenants/<slug>/` directory as authoritative. V3 reversed this to database-first with a single shared database. V4 keeps database-first but replaces the single shared database with a per-workspace isolation model.

### Mental model

When a user logs into ScottyDev and creates the "ScottyLand" workspace, all configuration is stored in the database immediately. However, v4's "database" means a *dedicated Postgres database* for that workspace, not a row in a shared database. The workflow is:

1. User logs into ScottyDev, authenticates via the existing auth chain.
2. User clicks "New Workspace," types a name, and hits Create.
3. ScottyDev (running as `scottydev:scottydev`) creates a row in the **admin database** (`scottydev_admin`): one row in `scottydev_workspaces` with a slug and a `linux_uid`.
4. ScottyDev provisions the workspace's isolated resources:
   - Creates a Linux user: `useradd --uid <linux_uid> --system --no-create-home ws-<slug>`
   - Creates the workspace data directory: `mkdir -p /var/scottydev/workspaces/<slug>`, `chown ws-<slug>:<slug> ...`, `chmod 0700 ...`
   - Creates a dedicated Postgres database: `CREATE DATABASE scottydev_ws_<slug>`, creates a Postgres role `scottydev_ws_<slug>` with access only to that database
   - Stores the workspace DB connection string (encrypted) in the admin database row
5. User navigates to workspace settings and fills in (stored in the **workspace's own DB**, not the admin DB):
   - Framework repo URL and credentials
   - Infrastructure connector — via AI-required wizard (see Q9)
   - AI backend config
6. User adds apps to the workspace project registry.

There is no `tenant.yaml`. Secrets never touch the filesystem except through the `.secrets/workspace_encryption_key` file (see Q15). The workspace's own database is authoritative for that workspace's data.

### Two-database architecture

```
scottydev_admin (Postgres)
├── scottydev_workspaces      ← workspace registry, linux_uid, ws DB conn string (encrypted)
├── scottydev_users           ← global ScottyDev users
├── scottydev_user_roles      ← global admin / viewer assignments
└── scottydev_workspace_summary  ← replicated aggregate rows for dashboard (see Q6/isolation)

scottydev_ws_scotty_land (Postgres, owned by role scottydev_ws_scotty_land)
├── workspace_config          ← connector config (encrypted), AI config (encrypted)
├── workspace_projects        ← app registry
├── workspace_members         ← per-workspace RBAC
├── workspace_chat_sessions   ← AI chat history
├── workspace_chat_messages
└── workspace_audit_log       ← immutable action log for this workspace

scottydev_ws_brians_projects (Postgres, owned by role scottydev_ws_brians_projects)
├── [same schema as above]
```

### Optional: local Claude Code overlay

When a developer wants to run a Claude Code session against the workspace's framework repo on their workstation, they can export a manifest:

```
GET /api/v1/workspaces/{slug}/local-overlay
```

This returns a `scottydev-workspace.yaml` (secrets redacted, replaced by env-var references) suitable for placing at the repo root. Claude Code reads it for context. This is a secondary, read-only projection of database state — not the source of truth.

### Feature flags per workspace

Feature flags are stored as a JSON column in the workspace's own database (in `workspace_config`), not in a `features.yaml` file. The Pydantic schema:

```python
class WorkspaceFeatures(BaseModel):
    project_registry: bool = True
    deploy_helper: bool = True
    dashboard: bool = True
    shared_core: bool = False
    pattern_tracking: bool = False
    sync_watcher: bool = False
    promote_scan: bool = False
    core_upgrade: bool = False
    release_fanout: bool = False
    manager_agents: bool = False
    ai_routing_rules: bool = False
    ai_chat: bool = True

    model_config = ConfigDict(extra="ignore")
```

Feature flag edits are a `PATCH /api/v1/workspaces/{slug}/features` call, requiring `admin` role on the workspace. Changes take effect immediately — no restart required.

## Q4 — Is ScottyDev multi-tenant SaaS, single-tenant, or something else? *(updated for v4 isolation model)*

V4 is honest: ScottyDev is a **self-hostable multi-tenant platform with OS-level workspace isolation that also supports a hosted deployment**.

A single ScottyDev install hosts N workspaces with separate Postgres databases, separate Linux UIDs, separate data directories, and separate encrypted credentials. In this user's case that is "ScottyLand" and "Brian's Projects."

Explicit properties:

- **Self-hostable.** One ScottyDev instance on a single Proxmox LXC or a Docker Compose stack. No cloud required.
- **Hosted-compatible.** The per-workspace isolation model makes a hosted deployment architecturally sound. Billing and self-service signup are not in scope for Phase 1.
- **Per-workspace compute isolation.** Each workspace gets its own Linux UID, Postgres database, and data directory. The ScottyDev process holds only a narrow admin credential. Connector operations run as the workspace's UID.
- **Not commercial SaaS today.** No billing, no self-service signup, no SLA. But the architecture supports it without changes.

## Q5 — What are the ScottyDev workspace capabilities? *(unchanged from v3)*

| Flag | Default | Description |
|---|---|---|
| `project_registry` | on | CRUD for tracked apps and repos |
| `deploy_helper` | on | Connector-driven deploy trigger, log streaming |
| `dashboard` | on | Overview page (feature-gated sections) |
| `ai_chat` | on | Conversational AI interface with workspace context (see Q16) |

| `sync_watcher` | off | Automated cross-app fix propagation |
| `promote_scan` | off | Manager agent GREEN/YELLOW/RED classification |
| `core_upgrade` | off | Pull new pattern versions from framework repo |
| `release_fanout` | off | Tag and release across all apps in registry |
| `manager_agents` | off | Sub-agent dispatch for UX/PM/DEV tasks |
| `ai_routing_rules` | off | Per-workspace AI backend routing overrides |

Four flags are always-on: `project_registry`, `deploy_helper`, `dashboard`, `ai_chat`. The remaining eight are opt-in. All eight pattern-discipline flags require `shared_core: true`; enabling them without `shared_core` produces a validation error.

## Q6 — Where does ScottyDev live in the codebase? *(updated for v4 isolation model)*

ScottyDev is a **separate repository** from `scottycore`. Following Shape C:

```
scottydev/                          ← new repo (scottydev on Forgejo/GitHub)
  scottydev/
    core/
      config.py                     # Pydantic Settings (DATABASE_URL, SECRET_KEY, etc.)
      database.py                   # SQLAlchemy async engine — admin DB + per-workspace DB factory
      auth.py                       # Reuses scottycore auth chain (imported as library)
      encryption.py                 # Envelope encryption for workspace credentials
      isolation.py                  # Linux UID management, workspace DB provisioning, privilege drop
    services/
      workspaces/                   # Workspace CRUD, feature flags, credential management
      connectors/                   # ConnectorBase + built-in connectors + AI wizard
      ai_chat/                      # Conversational AI interface (see Q16)
      projects/                     # App registry per workspace
      sync_watcher/                 # Cross-app fix propagation
      pattern_tracker/              # Drift detection
    api/v1/                         # FastAPI routes
    web/
      templates/                    # Jinja2 + HTMX + Alpine.js (same stack as scottycore apps)
    web/static/
  scripts/
    scottycore-init.py              # Moved from scottycore/scripts/ — now connector-aware
  .secrets/
    workspace_encryption_key        # Mode 0600, owned by scottydev process user — gitignored
  .gitignore                        # Includes .secrets/
  pyproject.toml                    # depends on scottycore as a library
```

`scottydev` depends on `scottycore` as a pip package:

```toml
[project]
name = "scottydev"
dependencies = [
    "scottycore @ git+https://forgejo.scotty.consulting/scotty/scottycore.git@v1.x",
    ...
]
```

ScottyDev gets the auth chain, RBAC, settings KV, audit log, and AI backends service by importing the library — it does not duplicate them.

### What moves out of scottycore

| What | Moves to |
|---|---|
| `scottycore/meta/` (v2 proposal, not yet built) | `scottydev/services/workspaces/` |
| `scripts/scottycore-init.py` | `scottydev/scripts/scottycore-init.py` (connector-aware rewrite) |
| `.claude/agents/*-manager.md` | `scottydev/.claude/agents/` |
| `config/apps.yaml` | Replaced by `workspace_projects` rows in each workspace's DB |
| `tenants/<slug>/` directories (v2, never built) | Never built; replaced by database |

### What stays in scottycore

Everything currently in `scottycore/scottycore/` stays. The Python package is unchanged. All six apps continue to work without modification.

## Q7 — What is the migration path? *(updated for v4 isolation provisioning)*

Three phases.

**Phase 1 — Foundation: connector abstraction + isolation model + repo split preparation**

- Create the `scottydev` repo on Forgejo and GitHub.
- Define `ConnectorBase` interface and implement the Phase 1 built-in connectors.
- Implement `scottydev/core/isolation.py`: Linux UID allocation, workspace DB provisioning, privilege-drop helpers.
- Rewrite `scottycore-init.py` to call `ConnectorBase` methods.
- Create `scottydev/services/workspaces/` with `WorkspaceService` writing to the admin DB and provisioning per-workspace DBs on create.
- Implement `scottydev/core/encryption.py` for envelope encryption.
- Implement the install setup wizard (CLI or web) that generates and persists the master key.
- All six consumer apps: zero changes required.

**Phase 2 — ScottyDev web UI (read-only AI chat)**

- Launch `scottydev` as a FastAPI app.
- Workspace creation flow: login → new workspace → framework repo → AI connector wizard → AI backend config.
- Project registry: add apps to workspace via UI.
- Deploy helper: trigger deploy via connector from the UI, stream logs via WebSocket.
- `ai_chat` feature: read-only tool set (see Q18).
- Dashboard with feature-gated sections.
- Global admin sees workspace list drawn from admin DB summary table; workspace admins see only their workspace.

**Phase 3 — Write tools + pattern discipline**

- Add write AI chat tools with confirmation UX and `scottydev_pending_confirmations` table (see Q18).
- Enable `shared_core`, `pattern_tracking`, `sync_watcher`, `promote_scan` for ScottyLand.
- Brian's Projects workspace remains at defaults.
- Release fan-out and manager agents.

## Q8 — What does ScottyDev actually do? *(updated for v4 isolation model)*

ScottyDev's four primary functions remain identical to v3 in intent. The v4 update is the execution context for deploy orchestration.

### 1. Dashboard (overview)

A single-page Jinja2 + HTMX view at `/`. Always rendered; each section is conditional on the active workspace's feature flags. The global admin's dashboard shows a workspace-list view populated from `scottydev_admin.scottydev_workspace_summary` (replicated aggregate data — no cross-workspace DB queries at runtime). A workspace admin's dashboard shows only their workspace.

### 2. Deploy orchestration (runs as workspace UID)

The user selects an app from the project registry and clicks Deploy. ScottyDev:

- Resolves the workspace's configured connector (loaded from the workspace's own DB).
- The ScottyDev request handler, running as `scottydev:scottydev`, drops privileges to the workspace's Linux UID before invoking the connector.
- Calls `connector.deploy_app(app_slug, branch)` in the workspace UID context.
- Streams stdout/stderr back to the browser via the existing `TaskEngine` WebSocket.
- Records the deploy event in the workspace's own `workspace_audit_log`.

The connector handles the actual SSH call or Docker operation. The workspace UID's filesystem access is limited to `/var/scottydev/workspaces/<slug>/`.

### 3. AI-assisted development (ai_chat)

See Q16 and Q18 for full model. Phase 2 is read-only; Phase 3 adds write tools.

### 4. Pattern discipline (shared_core workspaces)

Sync watcher, drift reports, promote-scan, release fan-out. These are the v2 features, unchanged in concept, now gated by the `shared_core` workspace flag and executing in the workspace's isolated context.

### Route-level feature gating

```python
@router.post("/api/v1/workspaces/{slug}/deploy")
async def trigger_deploy(
    slug: str,
    payload: DeployRequest,
    ws: ScottyDevWorkspace = Depends(get_active_workspace),
    features: WorkspaceFeatures = Depends(get_workspace_features),
    _: None = Depends(require_workspace_feature("deploy_helper")),
):
    ...
```

`require_workspace_feature()` returns HTTP 403 with `{"detail": "feature deploy_helper not enabled for workspace scotty-land"}`. It never returns 404.

## Q9 — Connector system architecture *(rewritten for v4: AI-required setup, generic naming)*

### The problem it solves

Most ScottyDev users will not have their own automation repo. They need a path to provision an LXC, deploy a Docker container, or configure an nginx vhost without writing Ansible playbooks first. V3's AI-wizard was an optional "I have existing workflows" path. V4 makes the AI wizard **the only path** — there is no click-through default for any connector.

### ConnectorBase interface *(unchanged from v3)*

Every connector must implement these six operations:

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class ProvisionResult:
    host_ip: str
    host_id: str
    notes: str = ""

@dataclass
class DeployResult:
    success: bool
    log_lines: list[str]
    url: str = ""

@dataclass
class HealthResult:
    reachable: bool
    http_status: int | None
    detail: str = ""


class ConnectorBase(ABC):
    """Abstract base for all ScottyDev infrastructure connectors.

    A connector is instantiated per-workspace from the workspace's
    decrypted connector_config JSON. It must be stateless between calls —
    all state lives in the database or on the target infrastructure.
    """

    @abstractmethod
    async def provision_app(self, app_slug: str, config: dict) -> ProvisionResult: ...

    @abstractmethod
    async def deploy_app(self, app_slug: str, repo_url: str, branch: str, port: int) -> DeployResult: ...

    @abstractmethod
    async def configure_dns(self, fqdn: str, target_ip: str) -> bool: ...

    @abstractmethod
    async def configure_tls(self, fqdn: str, apex: str) -> bool: ...

    @abstractmethod
    async def destroy_app(self, app_slug: str) -> bool: ...

    @abstractmethod
    async def health_check(self, app_slug: str, fqdn: str) -> HealthResult: ...

    @classmethod
    @abstractmethod
    def connector_type(cls) -> str: ...

    @classmethod
    @abstractmethod
    def config_schema(cls) -> type[BaseModel]: ...
```

### Built-in generic connectors

ScottyDev ships three built-in connectors. These connectors know nothing about "scottylab" or any specific user's infrastructure. They are generic building blocks the AI wizard selects from and parameterises.

**`ProxmoxSSHConnector`** (`connector_type = "proxmox_ssh"`)

Target: Proxmox homelab with LXC containers, nginx reverse proxy, Docker inside containers.

```python
class ProxmoxSSHConfig(BaseModel):
    proxmox_host: str
    proxmox_api_token: str
    nginx_host: str
    ssh_key_path: str
    ct_template: str = "local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst"
    domain_apex: str
    port_base: int = 8100
    cloudflare_token: str | None = None
```

**`DockerSSHConnector`** (`connector_type = "docker_ssh"`)

Target: a single Linux host accessible via SSH, running Docker Compose.

```python
class DockerSSHConfig(BaseModel):
    host: str
    ssh_user: str = "root"
    ssh_key_path: str
    deploy_base_path: str = "/opt"
    dns_provider: str = "cloudflare"
    dns_token: str
```

**`SystemdConnector`** (`connector_type = "systemd"`)

Target: a Linux host where apps run as systemd units (no Docker). Suitable for bare-metal, single-process Python or Node apps.

```python
class SystemdConfig(BaseModel):
    host: str
    ssh_user: str
    ssh_key_path: str
    service_name_template: str = "{app_slug}"
    deploy_base_path: str = "/opt"
    dns_provider: str = "cloudflare"
    dns_token: str
```

**`ExternalRepoConnector`** (`connector_type = "external_repo"`)

For users who have existing infrastructure automation — Ansible playbooks, Terraform modules, Jenkins pipelines, custom shell scripts — living in a Git repo. This connector clones the repo and invokes declared entry points. The connector itself knows nothing about "scottylab" or any specific tool set; it is generic. The AI wizard reads the repo's contents and generates the entry-point manifest.

```python
class ExternalRepoConfig(BaseModel):
    repo_url: str                        # Git URL of the automation repo
    repo_ref: str = "main"               # branch or tag to clone
    entry_points: dict[str, str]         # maps ConnectorBase op → file:function or shell command
    # e.g. {"deploy_app": "playbooks/deploy.yml:run", "health_check": "scripts/health.sh"}
    extra_vars: dict[str, str] = {}      # additional vars injected at runtime
    ssh_key_path: str | None = None      # if the automation repo or target needs a key
```

The user's scottylab repo is one example of what an `ExternalRepoConnector` points at. The connector itself is generic.

### AI-required connector setup wizard — no click-through path exists

This is non-negotiable. There is no "click a button and it works" factory default. Every workspace connector is configured through the AI wizard.

**Path A — User has existing automation (e.g., an Ansible repo, a scottylab-style toolkit)**

1. User selects "I have existing automation" in the connector setup UI.
2. User provides the Git URL of their automation repo (or uploads a tarball).
3. ScottyDev clones the repo into the workspace's isolated data directory on the ScottyDev host:
   `/var/scottydev/workspaces/<workspace-slug>/connector-repo/`
   This clone runs as the workspace's Linux UID (see Q6 isolation model).
4. An AI agent (using the workspace's configured AI backend — see AI backend options below) reads through the entire repo:
   - Lists the directory structure.
   - Reads key files (Makefiles, READMEs, entry scripts, playbooks).
   - Asks clarifying questions through the UI chat pane: "I see a `playbooks/deploy.yml` — does this handle both initial provisioning and re-deploys?" etc.
5. The AI generates a static `ExternalRepoConfig` manifest declaring the entry points for each `ConnectorBase` operation.
6. The generated manifest is saved to the workspace's database row (in the workspace's own DB), not the filesystem.
7. Subsequent connector invocations pull the external repo to `/var/scottydev/workspaces/<slug>/connector-repo/` and execute the declared entry points — running as the workspace's Linux UID.

**Path B — User has no existing automation**

1. User selects "I don't have existing automation, guide me through setup."
2. The AI agent asks (through the UI, chat-style):
   - "Where will this app be deployed?" Options: local bare-metal, local Docker Compose, LXC + Docker Compose, Azure Container Apps + Docker, AWS ECS, Kubernetes, bare-metal systemd, other (custom).
   - "Do you have existing repositories that should track infrastructure changes?"
   - "Do you have existing CI/CD workflows (GitHub Actions, Forgejo Actions, Jenkins, etc.) you want to leverage?"
   - Additional questions derived from earlier answers:
     - If local Docker Compose: "Which directory should hold the app? Any shared Docker network?"
     - If Azure ACA: "Subscription ID? Resource group? Container registry URL?"
     - If LXC + Docker Compose: "Proxmox host IP? API token? Domain apex?"
3. Based on answers, the AI selects and fully parameterises one of the built-in connectors: `ProxmoxSSHConnector`, `DockerSSHConnector`, or `SystemdConnector`.
4. If the user picks an unusual or unsupported target, the AI generates a `CustomPythonConnector` stub (a Python file extending `ConnectorBase`) that the user can review and edit before saving.
5. The resulting config is saved to the workspace's own database and the connector is tested via `health_check()` before the wizard closes.

**AI backend options for the wizard**

The wizard runs on whichever AI backend the workspace has configured. This is the workspace's `ai_config` (stored in the workspace's own DB). ScottyDev's existing `ai_backends` service from scottycore handles multi-provider routing; adding a new provider to scottycore's `ai_backends` automatically makes it available to the wizard.

Supported backends:
- **Claude Code CLI** — logged in on the ScottyDev host; uses the admin's Claude Code OAuth auth; no API key needed; always available on dev machines
- **Local GPU Ollama** — e.g., CT 114 at `192.168.150.210:11434`
- **DGX Ollama** — e.g., DGX Spark at `192.168.150.111:11434`
- **Claude API** — via `ANTHROPIC_API_KEY` in workspace AI config
- **OpenAI / Azure OpenAI** — via `OPENAI_API_KEY` or Azure endpoint + key
- **Any OpenAI-compatible API** — vLLM, TGI, LiteLLM; configured with a base URL

The first time a workspace is created, ScottyDev needs an AI backend configured before the connector wizard can run. The setup flow therefore asks for AI backend configuration before connector configuration.

### Connector registry

```python

    "proxmox_ssh":    ProxmoxSSHConnector,
    "docker_ssh":     DockerSSHConnector,
    "systemd":        SystemdConnector,
    "external_repo":  ExternalRepoConnector,
```

### Why per-workspace database, not schema-with-RLS?

Three options were evaluated:

**Option A — Database per workspace (recommended)**
- Each workspace gets a dedicated Postgres database and a dedicated Postgres role with access only to that database.
- Postgres enforces isolation at the connection level. A connection authenticated as `scottydev_ws_scotty_land` cannot even see the `scottydev_ws_brians_projects` database in `pg_catalog` without superuser.
- Alembic migration story: run Alembic against each workspace DB independently. The migration script iterates over admin DB rows to discover workspace DB connection strings, then migrates each. Operationally heavier but mechanically sound.
- Blast radius of a compromised Postgres admin credential: the `scottydev` process user holds a Postgres admin role (`scottydev_admin`) capable of accessing all workspace DBs for workspace CRUD (create database, create role, drop database). If this credential leaks, all workspace DBs are reachable. Mitigation: the admin role is used exclusively for workspace provisioning and deletion; API request handlers open workspace-scoped connections using the workspace's dedicated role, never the admin role.
- Backup: each workspace DB is backed up independently. pg_dump per workspace, scheduled separately. Adds operational overhead proportional to N workspaces.

**Option B — Schema per workspace with RLS**
- Single Postgres database, one schema per workspace, Row-Level Security policies on every table.
- Alembic story is simpler: one migration applies to all schemas.
- RLS is a policy layer, not a connection-level barrier. A bug in the RLS policy or a misconfigured `search_path` can leak data across schemas. Database-per-workspace prevents this class of bug entirely.
- Not recommended for v4 given the user's explicit "ZERO chance" requirement.

**Option C — Separate Postgres instance per workspace**
- Maximum isolation but operationally prohibitive on a homelab host. Not recommended.

**Decision: Option A (database per workspace).** The operational overhead is manageable at this scale (two workspaces initially). The isolation guarantee is stronger than RLS. The Alembic iteration script is a one-time engineering investment.

## Q14 — Per-workspace OS isolation model *(new in v4)*

This section answers the six sub-questions raised in Directive 6.

### Sub-question 1: Per-workspace Postgres database vs schema-with-RLS

**Recommendation: database per workspace.** See Q12 rationale. The one-database-per-workspace model prevents cross-workspace leakage at the Postgres connection-authentication level, not merely at the policy level. The migration overhead is acceptable.

### Sub-question 2: Linux user allocation strategy

**Recommendation: fixed UID range tracked in the admin DB.**

- Workspace Linux UIDs are allocated from the range `60000–69999` (well above system users, below typical uid_max on Linux hosts).
- The `scottydev_workspaces.linux_uid` column in the admin DB is authoritative.
- UID allocation: `INSERT INTO scottydev_workspaces (linux_uid, ...) VALUES (nextval('ws_uid_seq'), ...)` where `ws_uid_seq` starts at 60000.
- The OS user is created as: `useradd --uid <linux_uid> --system --no-create-home --shell /usr/sbin/nologin ws-<slug>`
- The workspace data directory: `mkdir -p /var/scottydev/workspaces/<slug> && chown ws-<slug>:nogroup /var/scottydev/workspaces/<slug> && chmod 0700 /var/scottydev/workspaces/<slug>`
- On workspace deletion (hard delete): `userdel ws-<slug>` and `rm -rf /var/scottydev/workspaces/<slug>` — only after the soft-delete retention period expires.

Using the slug-hash approach (derive UID from a hash of the slug) was considered and rejected: hash collisions require collision-resolution logic, and the admin DB is the right place to track system-level resource allocation.

### Sub-question 3: Subprocess-as-user vs Docker-per-connector-call

**Recommendation: subprocess-as-user (via `sudo -u`) for Phase 2, with Docker-per-connector-call as an optional hardened mode for Phase 3.**

Subprocess-as-user:
- The ScottyDev process user (`scottydev:scottydev`) is given a sudoers rule: `scottydev ALL = (ws-*) NOPASSWD: /usr/bin/python3 /var/scottydev/connector-runner.py *`
- The connector runner script is a fixed, audited Python script that receives the connector type and serialised config as arguments, instantiates the connector, runs the requested operation, and returns structured JSON output on stdout.
- The ScottyDev request handler calls: `subprocess.run(["sudo", "-u", f"ws-{slug}", "/usr/bin/python3", "/var/scottydev/connector-runner.py", ...])` and reads the JSON result.
- The workspace UID can access `/var/scottydev/workspaces/<slug>/` but not other workspace directories (OS enforces this).
- Tradeoff: a compromised connector could escape the Python subprocess and run arbitrary commands as `ws-<slug>`. It cannot read other workspaces' files (UID isolation), but it could exfiltrate its own workspace's data or make network calls.

Docker-per-connector-call:
- Each connector invocation spawns a short-lived Docker container from a fixed base image, with the workspace's data directory bind-mounted and credentials injected as environment variables.
- Harder to escape (container + UID isolation), but adds ~500ms latency per connector call and requires Docker on the ScottyDev host.
- Better suited for hosted/multi-tenant deployments where the workspace owner is an untrusted third party.

For this user's homelab (trusted workspaces, known owners), subprocess-as-user is appropriate. For a hosted product, Docker-per-connector-call should be the default. The architecture supports both; the choice is a config flag (`CONNECTOR_ISOLATION_MODE=subprocess|docker`).

### Sub-question 4: Meta-queries spanning workspaces (global admin dashboard)

**Recommendation: replicate aggregate data to the admin DB.** Querying each workspace DB in turn with admin credentials violates isolation and adds latency proportional to N workspaces.

The `scottydev_workspace_summary` table in the admin DB (see Q12) holds aggregate rows:

```python
class ScottyDevWorkspaceSummary(Base):
    __tablename__ = "scottydev_workspace_summary"
    workspace_id: Mapped[UUID]
    app_count: Mapped[int]
    last_deploy_at: Mapped[datetime | None]
    last_deploy_status: Mapped[str | None]
    health_status: Mapped[str | None]
    updated_at: Mapped[datetime]
```

Replication mechanism: when a deploy completes or a health check runs for workspace X, the workspace's request handler (which is running with workspace X's DB connection) emits a lightweight "summary update" event. A background task in ScottyDev (running as `scottydev:scottydev` with admin DB access) processes the event queue and writes the updated row to `scottydev_workspace_summary`. The workspace never writes to the admin DB directly — the background task holds the admin connection.

This means the global admin dashboard may be slightly stale (up to ~30 seconds). That is acceptable for a status dashboard.

### Sub-question 5: Global admin visibility vs workspace admin visibility

**Global admin** (ScottyDev `is_global_admin = true`):
- Sees the workspace list (drawn from `scottydev_workspace_summary`)
- Can create, rename, archive, and delete workspaces
- Can rotate workspace credentials
- Can manage global ScottyDev users
- **Cannot see workspace-internal data** (projects, chat history, connector config) unless explicitly added as a member of a specific workspace by its owner

**Workspace admin** (`role = owner` or `admin` in `workspace_members`):
- Sees only their workspace
- Cannot discover other workspaces exist (the workspace list API returns only workspaces they are a member of)
- Cannot see other workspaces' projects, connectors, or chat history

This is enforced at two levels:
- API: `get_workspaces_for_user()` filters by `workspace_members.user_id` for non-global-admins
- Database: workspace admin's Postgres role has access only to their workspace's DB; even if the API were bypassed, a direct Postgres connection as `scottydev_ws_scotty_land` cannot read `scottydev_ws_brians_projects`

### Sub-question 6: Workspace deletion retention

Soft-delete + hard-delete with 30-day retention:

1. **Soft delete**: Admin calls `DELETE /api/v1/workspaces/{slug}`. ScottyDev sets `is_deleted = true`, `deleted_at = now()`, `hard_delete_after = now() + 30 days` in the admin DB. The workspace is immediately hidden from all UI and API responses.
2. **During retention period (0–30 days)**: The workspace DB and `/var/scottydev/workspaces/<slug>/` directory remain intact. A global admin can restore the workspace (`PATCH /api/v1/admin/workspaces/{slug}/restore`).
3. **Hard delete (at 30 days)**: A nightly job checks `hard_delete_after <= now()` and executes:
   - `DROP DATABASE scottydev_ws_<slug>` (Postgres)
   - `DROP ROLE scottydev_ws_<slug>` (Postgres)
   - `userdel ws-<slug>` (OS)
   - `rm -rf /var/scottydev/workspaces/<slug>/` (filesystem)
   - Delete the admin DB row

The 30-day period is configurable via a global settings key `workspace_hard_delete_after_days`.

## Q15 — Master encryption key and workspace credential storage *(rewritten for v4)*

### Master key lifecycle

`WORKSPACE_ENCRYPTION_KEY` is a 32-byte random value, base64-encoded. It is **not** auto-generated by ScottyDev at startup.

**Generation and initial persistence:**
During the install setup wizard (the "install ScottyDev on my server" flow), the setup wizard:
1. Generates a cryptographically random 32-byte key: `key = base64.urlsafe_b64encode(os.urandom(32)).decode()`
2. Displays the key **once** in the wizard UI with the message: *"Save this key in your password manager now. It will not be shown again. Also provide it to your Claude Code session so AI tooling can work with encrypted workspace data."*
3. Creates `.secrets/` at the scottydev repo root and writes the key to `.secrets/workspace_encryption_key`, mode 0600, owned by the `scottydev` process user.
4. Adds `.secrets/` to `.gitignore`.

**Startup behaviour:**
- If `.secrets/workspace_encryption_key` exists: ScottyDev reads it and loads the key into `settings.workspace_encryption_key`. Startup proceeds normally.
- If `.secrets/workspace_encryption_key` does not exist: ScottyDev refuses to start and prints:
  ```
  ERROR: .secrets/workspace_encryption_key not found.
  ScottyDev cannot start without the master encryption key.
  Recovery: copy the key from your password manager into .secrets/workspace_encryption_key
  and set file permissions: chmod 0600 .secrets/workspace_encryption_key
  ```

**Key file path:** `<scottydev-repo-root>/.secrets/workspace_encryption_key`
**Mode:** 0600, owned by the `scottydev` process user.
**gitignore:** `.secrets/` is in `.gitignore` from day one.

For a hosted deployment, the key may alternatively come from a secret manager (Vault, AWS Secrets Manager) by overriding the startup logic to fetch from the manager instead of the file. The code path is the same — `settings.workspace_encryption_key` is populated regardless of source.

### Envelope encryption *(unchanged from v3)*

```
┌─────────────────────────────────────────────────────────────┐
│  WORKSPACE_ENCRYPTION_KEY (32 bytes, from .secrets/ file)   │
│  Loaded once at startup into settings.workspace_encryption_key
└─────────────────────────────────────────────────────────────┘
         │
         │  used to encrypt/decrypt
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Per-workspace data key (32 bytes random, per workspace)    │
│  Stored as: workspace_config.data_key_enc (encrypted with   │
│  master key, in the workspace's own DB)                     │
└─────────────────────────────────────────────────────────────┘
         │
         │  used to encrypt/decrypt
         ▼
┌─────────────────────────────────────────────────────────────┐
│  connector_config_enc  (bytes in workspace_config table)    │
│  ai_config_enc         (bytes in workspace_config table)    │
└─────────────────────────────────────────────────────────────┘
```

Cipher: AES-256-GCM via `cryptography` package. Fresh 12-byte nonce prepended to each ciphertext.

```python
def encrypt_json(data: dict, key: bytes) -> bytes:
    nonce = os.urandom(12)
    plaintext = json.dumps(data).encode()
    ciphertext = AESGCM(key).encrypt(nonce, plaintext, None)
    return nonce + ciphertext

def decrypt_json(blob: bytes, key: bytes) -> dict:
    nonce, ciphertext = blob[:12], blob[12:]
    plaintext = AESGCM(key).decrypt(nonce, ciphertext, None)
    return json.loads(plaintext)

def _load_master_key() -> bytes:
    key_path = Path(settings.scottydev_root) / ".secrets" / "workspace_encryption_key"
    return base64.urlsafe_b64decode(key_path.read_text().strip())
```


This iterates over all workspace DBs (discovered via admin DB), re-encrypts all `*_enc` columns, writes the new key file, and prints a summary. Old key file is not deleted by the tool — the operator should do so manually after verifying.

### Credential injection into connectors *(unchanged from v3)*

When `WorkspaceConnectorLoader.load()` instantiates a connector, it passes the decrypted config dict directly to the connector's `config_schema()` constructor. The decrypted dict exists only in memory for the duration of the request. It is never logged, returned via API, or written to disk.

## Q16 — AI-in-the-loop for orchestration *(updated for per-workspace AI backend; tool scope deferred to Q18)*

## Q18 — Phase 2 → Phase 3 AI chat tool scope expansion *(new in v4)*

This section exists to track the read-only / write-tool split commitment. The planner agent will use this section to produce sprint tasks.

### Phase 2 AI chat tools (read-only)

These tools are available when `ai_chat` is enabled. They make no mutations to any repository or deployment.

| Tool | Signature | Description |
|---|---|---|
| `read_file` | `(repo: str, path: str, ref: str = "HEAD") → str` | Read a file from a project or framework repo via Forgejo API |
| `list_files` | `(repo: str, path: str = "/", ref: str = "HEAD") → list[str]` | List directory contents of a repo |
| `get_drift_report` | `(workspace_slug: str) → DriftReport` | Return current pattern drift report for this workspace |
| `run_health_check` | `(project_slug: str) → HealthResult` | Call `connector.health_check()` for a registered app |

### Phase 3 AI chat tools (write — added in Phase 3)

These tools make mutations. Each write tool call is intercepted before execution and creates a `scottydev_pending_confirmations` row. The UI shows the pending action to the user with a diff or summary. The action executes only after the user clicks "Approve." The user may also "Reject," which cancels the pending action.

| Tool | Signature | Description | Confirmation display |
|---|---|---|---|
| `commit_file` | `(repo: str, branch: str, path: str, content: str, message: str) → CommitResult` | Commit a file change to a repo branch via Forgejo API | Diff of old vs new content |
| `create_branch` | `(repo: str, branch_name: str, from_ref: str) → bool` | Create a new branch in a repo | Branch name + base ref |
| `trigger_deploy` | `(project_slug: str, branch: str) → task_id: str` | Trigger a deploy via the workspace connector | App name + branch; logs streamed after approval |
| `rollback_deploy` | `(project_slug: str, to_ref: str) → task_id: str` | Roll back a deployed app to a previous commit | App name + target ref |
| `update_app_config` | `(project_slug: str, changes: dict) → WorkspaceProject` | Update a project's registered config (port, FQDN, branch) | Field-by-field diff |

### `scottydev_pending_confirmations` table schema

This table lives in each workspace's own database (not the admin DB). It is only present in Phase 3.

```python
class ScottyDevPendingConfirmation(Base):
    """Tracks write tool calls awaiting user approval in the AI chat."""
    __tablename__ = "scottydev_pending_confirmations"
    id: Mapped[UUID] = mapped_column(primary_key=True, default=uuid4)
    session_id: Mapped[UUID]              # FK → workspace_chat_sessions.id
    tool_name: Mapped[str]                # e.g. "commit_file", "trigger_deploy"
    tool_args_json: Mapped[str]           # serialised tool arguments
    display_summary: Mapped[str]          # human-readable description shown in UI
    diff_json: Mapped[str | None]         # JSON-encoded diff for commit_file
    status: Mapped[str] = mapped_column(default="pending")
    # status values: pending | approved | rejected | expired | executed
    created_at: Mapped[datetime] = mapped_column(default=func.now())
    expires_at: Mapped[datetime]          # auto-expire after 10 minutes if no action
    resolved_at: Mapped[datetime | None]
    resolved_by_user_id: Mapped[UUID | None]
```

**Confirmation flow:**
1. AI requests a write tool call (e.g., `commit_file`).
2. ScottyDev intercepts the call, creates a `scottydev_pending_confirmations` row with `status = "pending"`.
3. Tool result returned to AI: `{"status": "pending_confirmation", "confirmation_id": "<uuid>", "message": "Awaiting user approval in the UI"}`.
4. AI's response to the user explains what it wants to do and that it is waiting for confirmation.
5. UI shows a confirmation card with the summary/diff. User clicks "Approve" or "Reject."
6. `POST /api/v1/workspaces/{slug}/confirmations/{id}/approve` or `.../reject`.
7. On approval: ScottyDev executes the tool call, updates the confirmation row to `status = "executed"`, continues the chat session with the tool result.
8. Confirmations that are neither approved nor rejected within 10 minutes are set to `status = "expired"`. If the user tries to approve an expired confirmation, they receive an error and must ask the AI to re-propose.

**The `scottydev_pending_confirmations` table is not present in Phase 2.** The Phase 2 tool set is read-only and requires no confirmation mechanism. The table is created by a Phase 3 Alembic migration applied to each workspace's own DB.

## Appendix B — What ScottyDev is NOT *(unchanged from v3)*

- Not a CI/CD system. It triggers deploys; it does not replace Forgejo CI or GitHub Actions.
- Not a secrets manager. Envelope encryption in the database is not the same as a rotating secrets vault.
- Not a monitoring system. It reads deploy status and health check responses; it does not replace Grafana or Prometheus.
- Not a code review tool. Manager agents classify PRs; they do not merge or block them.
- Not a full coding agent. The AI chat can read files and commit targeted changes via Forgejo API; it is not a browser-hosted Claude Code session with full filesystem access.
- Not locked to self-hosting. The architecture supports a hosted deployment without modification.

**Decision 1 (resolved in v3): scottylab_toolkit resolution.**

**Decision 2 (resolved in v3): Forgejo repo rename.**
Resolved: `scotty/scottycore` stays `scotty/scottycore`. A new `scotty/scottydev` repo is created.

**Decision 3 (resolved in v3): `scottycore-patterns` repo creation timing.**
Resolved: create in Phase 1.

**Decision 4 (resolved in v4): WORKSPACE_ENCRYPTION_KEY bootstrap.**
Resolved: the setup wizard generates the key, displays it once, and persists it to `.secrets/workspace_encryption_key` (mode 0600, gitignored). ScottyDev refuses to start if the file is absent. The admin owns the password-manager copy for recovery. See Q15.

**Decision 5 (resolved in v4): AI chat tool confirmation UX.**
Resolved: Phase 2 launches with read-only tools only (`read_file`, `list_files`, `get_drift_report`, `run_health_check`). Write tools (`commit_file`, `create_branch`, `trigger_deploy`, `rollback_deploy`, `update_app_config`) are added in Phase 3 with the `scottydev_pending_confirmations` table. See Q18.

**Decision 6 (new in v4 — resolved): Per-workspace isolation model.**
Resolved: database-per-workspace, dedicated Linux UID, subprocess-as-user for connector execution (with Docker isolation as optional Phase 3 hardened mode), aggregate summary replication to admin DB for global dashboard. See Q14.

**Decision 7 (new in v4 — open): Connector isolation mode for hosted deployment.**
For this user's homelab, `CONNECTOR_ISOLATION_MODE=subprocess` is appropriate. If ScottyDev is ever offered as a hosted product where workspace owners are untrusted third parties, `CONNECTOR_ISOLATION_MODE=docker` should be the default. **No action required for Phase 1 or 2, but the config flag should be added to `scottydev/core/config.py` in Phase 1 so the switch is available.**

**Decision 8 (new in v4 — open): Alembic multi-DB migration strategy.**
Each workspace DB must be migrated when the workspace schema changes. The migration runner must: (1) read workspace DB connection strings from the admin DB, (2) decrypt them, (3) run Alembic `upgrade head` against each. The design of this runner (CLI script vs Makefile target vs CI/CD step) needs to be specified before Phase 2 development begins. **Recommend a standalone CLI: `python -m scottydev.cli migrate-workspaces --confirm`.**

## Inline summary for parent agent

**Biggest v3 → v4 shift**

The single largest change is the replacement of the shared-database workspace model with a per-workspace Postgres database + Linux UID isolation model. V3's `workspace_id` column partitioning was a logical barrier; v4's model is a physical barrier enforced at the OS authentication and Postgres connection levels simultaneously. This affects the database schema, the startup provisioning sequence, the connector execution path (privilege drop), the migration runner, and the global admin dashboard (which now reads from a replicated summary table rather than querying workspace tables directly).

**Three firm recommendations on isolation sub-decisions**

1. **DB-per-workspace vs RLS**: Recommend database-per-workspace. RLS is a policy, not a connection-level barrier. A RLS misconfiguration can expose all workspaces; a misconfigured workspace Postgres role cannot reach a sibling database. The Alembic overhead is a one-time engineering investment, not ongoing complexity.

2. **UID allocation**: Recommend fixed range (60000–69999) tracked in the admin DB `scottydev_workspaces.linux_uid` column, allocated via a Postgres sequence. Slug-hash derivation introduces collision-resolution complexity with no benefit.

3. **Subprocess vs Docker**: Recommend subprocess-as-user (`sudo -u ws-<slug>`) for Phase 1/2 (homelab use case, trusted workspace owners, low latency). Add `CONNECTOR_ISOLATION_MODE=docker` as an optional mode in Phase 3 for hosted/untrusted deployments. The config flag should be in place from Phase 1 even if only `subprocess` is implemented initially.

**New decisions surfacing in v4 (not present in v2 or v3)**

- **Decision 7**: Connector isolation mode flag for hosted deployment — must be added to `config.py` in Phase 1, even if only `subprocess` mode is implemented initially.
- **Decision 8**: Multi-workspace Alembic migration runner strategy — must be designed before Phase 2 development begins, as every workspace schema change requires it.
- **Implicit new decision**: The AI connector wizard requires an AI backend to be configured before the connector can be set up. This means the workspace creation flow has a strict ordering dependency (AI backend first, then connector wizard). The UI setup flow must enforce this ordering and provide a clear error if a user tries to start the connector wizard before configuring an AI backend.
