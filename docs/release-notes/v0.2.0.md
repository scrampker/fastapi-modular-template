# scottycore v0.2.0 — BrandConfig: deploy-anywhere support

Minor version bump (backwards compatible defaults). Adds a single new
primitive — `scottycore.core.brand.BrandConfig` — that threads through
every place that previously hard-coded brand strings.

## What changed

### `scottycore.core.brand`

New module with a frozen `BrandConfig` dataclass, `get_brand()` singleton
accessor, and `reset_brand_cache()` for tests. All fields default to the
scotty values, so existing consumer apps see no behaviour change.

```python
from scottycore.core.brand import get_brand

brand = get_brand()
brand.framework_name       # "scottycore" (or "briancore" for a fork)
brand.family_name          # "scotty"
brand.orchestrator_name    # "scottydev"
brand.infra_worker_name    # "scottylab"
brand.infra_worker_url     # "" → solo mode, else a URL
brand.domain_root          # "scotty.consulting"
brand.framework_repo_url   # git URL for the framework
brand.display_name         # "Scotty"

# Derived:
brand.config_dir           # /etc/scottycore
brand.apps_root            # /opt/scottycore
brand.data_dir_env_var     # SCOTTYCORE_DATA_DIR
brand.systemd_unit_prefix  # scottycore-update-check
brand.update_mode_path     # /etc/scottycore/update-mode
brand.update_setting_key_mode     # scottycore.update.mode
brand.update_setting_key_pending  # scottycore.update.pending
brand.rebuild_flag_filename       # .scottycore-auto-update-requested
brand.pin_pattern          # regex for <framework> @ git+…@<ref>
```

Loaded from env on first call; forks override any of the above via
`BRAND_FRAMEWORK_NAME`, `BRAND_FAMILY_NAME`, etc.

### Auto-update service is brand-aware

`AutoUpdateService` now accepts `brand: BrandConfig | None` and derives:

* Pin regex (was hard-coded to `scottycore @ git+...@REF`)
* Settings keys (`<framework>.update.mode`, `<framework>.update.pending`)
* Rebuild flag filename (`.<framework>-auto-update-requested`)
* Default forge URL (from `brand.framework_repo_url`)

The CLI `prog` name follows the brand too — running `briancore-update-check`
(installed via a fork's entry point) shows `briancore` in `--help`.

### systemd unit templates

The old hard-coded `scottycore-update-check@.{service,timer}` files are
gone. Replaced with four Jinja templates under
`scottycore/services/auto_update/systemd/`:

* `update-check@.service.j2`
* `update-check@.timer.j2`
* `app-update-check@.service.j2`
* `app-update-check@.timer.j2`

Rendered at install time via the new
`scottycore.services.auto_update.systemd_render.render_systemd_units()`
helper or by scottydev's `deploy/install-brand.sh`.

### Data-dir env var follows the brand

`main.py` used to look up `SCOTTYCORE_DATA_DIR`. Now it reads
`brand.data_dir_env_var` — which is `SCOTTYCORE_DATA_DIR` by default but
becomes `BRIANCORE_DATA_DIR` in a brian fork.

### Backup CLI prog name is brand-aware

`scottycore-backup` CLI argparse `prog=` now uses `f"{brand.framework_name}-backup"`,
so forks shipping the same code under a different entry-point name get a
consistent experience.

## How to use in your app

**No action required for existing scotty-branded apps.** Defaults preserve
every previous name, path, and setting key.

**For forks:**

1. Set env vars before the app boots:

   ```bash
   export BRAND_FRAMEWORK_NAME=briancore
   export BRAND_FAMILY_NAME=brian
   export BRAND_ORCHESTRATOR_NAME=briandev
   export BRAND_INFRA_WORKER_NAME=brianlab
   export BRAND_INFRA_WORKER_URL=https://brianlab.brian.example.com
   export BRAND_DOMAIN_ROOT=brian.example.com
   export BRAND_FRAMEWORK_REPO_URL=https://github.com/you/briancore.git
   export BRAND_DISPLAY_NAME=Brian
   ```

2. Add the brand env vars to your systemd `EnvironmentFile` or Docker
   `env_file` so every process sees them.

3. Install the orchestrator's `deploy/install-brand.sh` to generate
   brand-prefixed systemd units + shell wrappers on the host.

That's it. Every path, env var, setting key, systemd unit name, and
update-notification follows the brand — a fork is a drop-in replacement.

## Compatibility

Backwards compatible. No migrations needed. Consumer apps on v0.1.10 can
bump to v0.2.0 freely; the only visible change without explicit brand
overrides is… none. Defaults are scotty.
