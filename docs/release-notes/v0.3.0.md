# scottycore v0.3.0 — White-label: 200% brand-agnostic

Minor bump, backwards-compatible defaults. Wave 2 of the BrandConfig
refactor — closes the gap that v0.2.0 left in user-visible surfaces so
a customer can deploy the stack and their end users never see "Scotty"
anywhere.

## What changed

### Jinja brand globals

`scottycore.web.setup_jinja_brand(env)` — drop-in helper that injects
`brand`, `brand_display_name`, and `orchestrator_display` as Jinja
globals on any `Environment`. The default `scottycore.web.router.templates`
instance gets these globals automatically.

Templates can now reference:

```jinja
{{ brand.display_name }}     → "Scotty" (or fork override)
{{ brand.domain_root }}      → "scotty.consulting"
{{ brand.framework_name }}   → "scottycore"
{{ orchestrator_display }}   → "ScottyDev" (derived)
```

### base.html exposes brand to JS

`templates/base.html` now injects `window.__BRAND__` and
`window.__BRAND_ORCHESTRATOR_DISPLAY__` into every page so Alpine/HTMX
logic can pick up the brand at runtime.

### admin_backups.html is brand-aware

Previously: 8 literal "ScottyDev" strings baked into Alpine bindings,
option text, and JS display maps.
Now: all reference `window.__BRAND_ORCHESTRATOR_DISPLAY__` or the Jinja
`{{ orchestrator_display }}` global.

### User-visible error messages use the brand

- `service.py` restore error: "upgrade scottycore" → "upgrade {brand.framework_name}"
- `scheduler.py` + `api/v1/backup.py`: "scottydev sink requires base_url"
  → "orchestrator sink requires base_url"
- `sinks/scottydev.py` HTTP error: "scottydev sink failed" → "orchestrator sink failed"
- `api/v1/backup.py` 409 conflict: "Managed by ScottyDev" → "Managed by {orchestrator}"

The underlying enum value (`managed_by = 'scottydev'` in the DB) stays for
schema backwards-compat. Only the user-visible string surfaces change.

### CLI help text follows the brand

`scottycore-backup --help` and `scottycore-update-check --help` already
used `brand.framework_name` for the prog name in v0.2.0. v0.3.0 finishes
the job:

- `--base-url` help: "scottydev sink base URL" → "orchestrator sink base URL"
- `--token` help: "scottydev sink bearer token" → "orchestrator sink bearer token"

### CI workflows are repo-var-driven

The `.forgejo/workflows/*.yml` files previously hardcoded
`https://forgejo.scotty.consulting`, `scotty/scottycore`, `scottycore-runner:latest`,
and `scottycore-bot`. v0.3.0 parametrizes these via Forgejo repository
variables with scotty defaults:

| Repo variable | Default | Drives |
|---|---|---|
| `FORGEJO_BASE` | `https://forgejo.scotty.consulting` | API base URL |
| `FORGEJO_ORG` | `scotty` | Org slug in dispatch URLs |
| `BRAND_FRAMEWORK_NAME` | `scottycore` | pip pin regex, upgrade workflow filename |
| `BRAND_RUNNER_IMAGE` | `scottycore-runner:latest` | Self-hosted runner image |
| `BRAND_BOT_NAME` | `scottycore-bot` | git commit author on bumps |
| `BRAND_BOT_EMAIL` | `bot@scotty.consulting` | git commit author email |
| `BRAND_UPGRADE_APPS` | `scottystrike scottybiz scottylab-app scottydev` | Dispatch recipients |

A fork sets these in their Forgejo repo/org settings — zero code edits
in the workflow files.

### Release URL uses `github.repository`

The release.yml release-creation curl previously hardcoded
`repos/scotty/scottycore/releases` — a latent bug on any fork. v0.3.0
uses `${GITHUB_REPOSITORY}` (populated by Forgejo as `<org>/<repo>`)
which is correct regardless of brand.

## How to use in your fork

```bash
# On each LXC / VM running a consumer app:
cat > /etc/brand.env <<'EOF'
BRAND_FRAMEWORK_NAME=briancore
BRAND_FAMILY_NAME=brian
BRAND_ORCHESTRATOR_NAME=briandev
BRAND_INFRA_WORKER_URL=https://brianlab.brian.example.com
BRAND_DOMAIN_ROOT=brian.example.com
BRAND_FRAMEWORK_REPO_URL=https://github.com/you/briancore.git
BRAND_DISPLAY_NAME=Brian
EOF

# In your Forgejo repo variables (UI: Settings → Actions → Variables):
FORGEJO_BASE=https://forgejo.brian.example.com
FORGEJO_ORG=brian
BRAND_FRAMEWORK_NAME=briancore
BRAND_RUNNER_IMAGE=briancore-runner:latest
BRAND_BOT_NAME=briancore-bot
BRAND_BOT_EMAIL=bot@brian.example.com
BRAND_UPGRADE_APPS=brianstrike brianbiz brianlab
```

## Compatibility

Fully backwards-compatible. Every default matches the scotty fleet
verbatim — `scotty/*` repos, `scottycore-bot`, `scottycore-runner:latest`,
"ScottyDev" as the orchestrator display — so zero change for existing
deployments.

Consumer apps on scottycore v0.2.0 can bump to v0.3.0 freely.

## What's still scotty

Honest list of what v0.3.0 does NOT change (still scotty-branded):

- **Python package name** `scottycore` — library name, not runtime config
- **Python package name** `scottydev` — same
- **Class names** like `ScottyLabClient`, `ScottyDevSink` — internal implementation
- **DB enum value** `managed_by = 'scottydev'` — schema-level, requires migration
- **DB table prefixes** (`scottydev_apps`, `scottydev_workspaces`) — schema-level
- **Docker image default** `scottydev:latest` — but now overridable via
  `${BRAND_ORCHESTRATOR_NAME:-scottydev}` in docker-compose.yml

A true fork-time rename (package + DB + class names) is a v0.4.0 concern.
For customer deployments, the runtime + UI + CI surface is now agnostic
— customers' users and admins see only the customer's brand.

## Tests

Full scottycore suite: 210 passed, 1 skipped (unchanged from v0.2.0 —
the refactor is display/config only).

## The fundamental model (unchanged from v0.2.0)

`BrandConfig` is a frozen dataclass, loaded once at startup via
`@lru_cache`. v0.3.0 adds more consumers of the same singleton —
Jinja globals, error messages, CI env vars — without touching the
underlying primitive.
