## Highlights

Additive release — two new service modules extracted from scottystrike:

- **`scottycore.services.sync`** — mesh sync subsystem (PlatformNode, SyncPeer, SyncTenantSubscription, SyncConflict, PeerApiKey). LWW conflict resolution, vault-based peer key encryption, bcrypt-prefix API key pattern. ~2000 lines across 8 files.
- **`scottycore.services.backup`** — tarball-based export/restore with a pluggable contributor pattern. ~1000 lines across 5 files. Contributors register for PLATFORM and/or TENANT scopes; restore iterates in registration order to respect FK dependencies.

No breaking changes. Apps that don't import these services continue unchanged.

## Adoption Guides

### Using the new `SyncService`

**Before** (scottystrike's vendored copy, hardcoded node name):
```python
# scottystrike/app/core/service_registry.py
from scottycore.services.sync.service import SyncService
sync = SyncService(session_factory, audit_service)
# node_name was hardcoded inside service.py as "scottystrike"
```

**After** (scottycore v0.1.2 — node name parameterized):
```python
from scottycore.services.sync.service import SyncService
# Default: reads settings.app_name
sync = SyncService(session_factory, audit_service)
# Explicit override (multi-node deploys, etc.):
sync = SyncService(session_factory, audit_service, node_name="my-node-01")
```

### Using the new `BackupService`

**Before** (scottystrike's vendored copy, hardcoded app metadata):
```python
# _APP_NAME + _APP_VERSION module constants baked into service.py
backup = BackupService(session_factory, audit_service)
```

**After** (scottycore v0.1.2 — app metadata parameterized):
```python
from scottycore.services.backup.service import BackupService
# Default: reads settings.app_name + getattr(settings, "app_version", "0.0.0")
backup = BackupService(session_factory, audit_service)
# Explicit override:
backup = BackupService(
    session_factory, audit_service,
    app_name="myapp", app_version="1.2.3",
)
```

## Migration required for adopters

If you were using the `scottystrike` vendored copy of these services, dropping your `scottycore/services/sync/` and `scottycore/services/backup/` trees in favor of the upstream pip package will:

1. Inherit the parameterized constructor (`node_name` / `app_name` / `app_version`) defaults from `settings.app_name`. No code changes required if your app correctly populates `SCOTTYCORE_APP_NAME` (or equivalent env var).
2. Not break your existing SQLAlchemy models or Alembic migrations — the models are byte-identical to scottystrike's vendored copy.

**One caveat on Alembic**: scottycore v0.1.2 does NOT ship a migration for the mesh sync tables. If your app doesn't already have migration `0005_mesh_sync_tables.py` (or equivalent), run `alembic revision --autogenerate` after upgrading; SQLAlchemy will detect the new tables from `scottycore.services.sync.models` and emit a migration automatically.

## What's NOT in this release

- No REST API routes for sync or backup — services only. Consumer apps wire their own `/api/v1/sync/*` and `/api/v1/backup/*` route layers. This is consistent with every other scottycore service.
- No ServiceRegistry auto-registration. Consumer apps instantiate these in their own service wiring (same pattern as `AuditService`, `TenantsService`, etc.).

## Provenance

Extracted from scottystrike — specifically the code under `scottystrike/scottycore/services/sync/` and `scottystrike/scottycore/services/backup/` at commit `03d8eae5`. The promote-scan classifier flagged these correctly; the automated promote-receive run aborted due to a workflow path bug (now patched with diagnostics in the same commit). Scottystrike retains authorship credit for the code; scottycore is adopting it as canonical.
