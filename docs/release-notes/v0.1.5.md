# scottycore v0.1.5 — Backup & Restore + install_static

Biggest release since v0.1.0. Two threads:

1. **Full backup/restore subsystem** — storage sinks, GPG encryption, REST
   API, CLI, scheduler, admin GUI.
2. **`install_static` helper** (carried over from v0.1.4 work) — one place
   for consumer apps to mount the scottycore static tree alongside their
   own overlay.

## Backup & Restore

### What's new

* **Storage sinks** (`scottycore.services.backup.sinks`) — `StorageSink` ABC
  plus four implementations:
  * `LocalDiskSink` — filesystem-backed, atomic rename writes,
    `.meta.json` sidecars, streaming verify, path-traversal guards
  * `ScottyDevSink` — HTTP PUT/GET/DELETE against ScottyDev's backup-store
    API with Bearer auth
  * `RemoteNodeSink` — ssh/scp shell-out (configurable port/key/options)
  * `DownloadSink` — ephemeral pass-through for browser downloads
* **GPG passphrase encryption** (`scottycore.services.backup.crypto`) —
  symmetric AES-256 + SHA-512 S2K via `gpg --symmetric`. `fingerprint()`
  helper so operators can match a saved passphrase to a snapshot without
  logging the passphrase itself.
* **Schema version gate** on restore — `UnsupportedBundleError` fires when
  a bundle's `schema_version` exceeds the build's `SUPPORTED_SCHEMA_VERSION`.
* **`BackupSchedule` + `BackupRun` models** with dual-management fields
  (`managed_by` / `remote_id` / `passphrase_override_fingerprint` /
  `key_fingerprint`). Alembic migration `0003`.
* **`FilesContributor`** — ships `uploads_base_dir` trees alongside DB rows,
  size-capped per file.
* **`SecretsContributor`** — pattern-classified settings keys (passwords,
  tokens, API keys, webhooks). Re-filters on restore to block trojaned
  rows under this contributor id.
* **REST API** under `/api/v1/backups/*`:
  * `GET  /contributors`
  * `POST /export` (routes to sink) and `POST /export/download`
  * `POST /restore` (from sink) and `POST /restore/upload`
  * `GET  /runs`, `GET  /schedules`, `POST /schedules`,
    `DELETE /schedules/{id}`
  * Sink-store endpoints (`PUT/GET/DELETE /store/{path}` + `GET /index`)
    so the app can serve as a target for `ScottyDevSink`
* **CLI** — `scottycore-backup {export,restore,verify,list-runs,rotate-key}`.
  Runs directly against the DB (no HTTP round-trip). Passphrase supports
  `-` for stdin / getpass.
* **Scheduler** (`scottycore.services.backup.scheduler`) — asyncio cron
  driver with retention (`keep_last` / `retention_days`), pluggable
  `PassphraseProvider`, serial dispatch, failed-run retry, one-shot
  auto-deactivation.
* **Admin GUI** at `/admin/backups` — PBS-style 4-tab console: Datastores /
  Jobs / Snapshots / Contributors. "New backup" modal streams to browser or
  pushes to a sink; Restore modal prompts for passphrase when snapshot is
  encrypted.

### Wiring contract

`ServiceRegistry.backup` is now populated automatically by
`build_backup_service(session_factory, audit_service, uploads_base_dir=...)`.
Consumer apps can register domain contributors on top:

```python
from scottycore.services.backup.schemas import BackupScope

registry.backup.register(MyDomainContributor(registry.session_factory))
```

Contributors registered in FK-safe order run first; apps can add more
after them. See `scottycore/services/backup/contributor.py` for the
Protocol.

### Dependency adds

* `croniter>=2.0` (scheduler)

### Environment

* `SCOTTYCORE_BACKUP_STORE` — root dir for blobs received via the sink-store
  endpoints. Defaults to `/app/data/backup-store`.

## install_static helper (from v0.1.4)

`scottycore.web.install_static(app, overlay_dir, mount_path, ...)` now
encapsulates mounting scottycore's static tree alongside an app overlay.
See `v0.1.4.md` for the full adoption guide.

## Upgrade path for consumer apps

1. Bump the pin in `pyproject.toml`:
   ```
   "scottycore @ git+https://github.com/scrampker/scottycore.git@v0.1.5"
   ```
2. Run `alembic upgrade head` (picks up `0003_backup_schedules_and_runs`).
3. (Optional) Replace manual StaticFiles mounts with
   `scottycore.web.install_static(app, overlay_dir="app/web/static")`.
4. (Optional) Register domain backup contributors on
   `registry.backup.register(...)` in your startup code.
